Your Role in One Line
You are the brain of Smartilee. You build the Flask server that understands every WhatsApp message from a
prospective student — what country they are interested in, which package they are asking about, how they
feel, what language they speak, and what their consultation history is — and generates a personalised reply
grounded in the study abroad consultancy's real package data using RAG. Every other component depends
on your endpoints working.
THE TWO BIG CONCEPTS YOU OWN
Concept 1 — RAG (Retrieval Augmented Generation)
The LLM on its own knows nothing about your consultancy's packages, pricing, or visa rules. RAG fixes this.
Before the LLM generates any reply, your system first searches the knowledge_base table in Supabase for
the most relevant rows — Germany tuition fees, France scholarship details, Netherlands visa process,
eligibility criteria, intake dates — and injects them into the LLM prompt as context. The LLM then answers
from your real package data, not from generic internet knowledge. The student gets an accurate, specific
answer every single time. The updated match_knowledge_base RPC accepts filter_country and
filter_category parameters — when a student specifies Germany, pass filter_country='Germany' to retrieve
only Germany-relevant rows.
Concept 2 — Ollama (Local LLM)
Groq runs the LLM on their cloud servers. Ollama runs the exact same LLM on your own machine — no API
key, no credits, no data leaving your infrastructure. For the hackathon demo you use Groq because it is
sub-second fast. But the Flask backend is architected so switching to Ollama is one environment variable
change. Same code, different base URL. This is your production path.

HOW THE RAG PIPELINE WORKS FOR STUDY ABROAD
Step 1
load_knowledge_base() at startup
Reads all rows from the knowledge_base table in Supabase (Germany/France/Netherlands package data,
visa rules, scholarship details, FAQs, intake dates). Converts each row's content into a vector embedding
and stores it in Supabase using pgvector.
Step 2
retrieve_context(message_text, country=None) on every message
Converts the student's message into a vector. If the student has a preferred_country in their profile (e.g.
'Germany'), calls match_knowledge_base with filter_country='Germany' for more precise results. Returns
the top 4 most semantically relevant knowledge base rows.
Step 3
Inject retrieved rows into the LLM prompt
The prompt says: 'Here is the relevant information from our study abroad packages: [rows]. Using ONLY this
information, answer the student's question in [language]. Do not invent fees, dates, or scholarship amounts
not in the context.'
Step 4
LLM generates reply and it goes back to the student
Groq LLaMA 3 generates a natural, accurate, personalised reply. Integration engineer sends it back via
WhatsApp through Happilee.
FLASK ENDPOINTS YOU MUST BUILD
POST /analyze
Input: { phone_number, message_text }
Output: { intent, sentiment, language, action, churn_score, risk_level, preferred_country }
Main endpoint called by the Integration engineer for every incoming student message. Runs the full pipeline: language
detection -> fetch customer profile (get preferred_country) -> RAG context retrieval (with filter_country if known) ->
intent detection -> sentiment analysis -> churn scoring -> decision engine.
POST /generate-reply
Input: { phone_number, message_text, intent, sentiment, language }
Output: { reply_text }
Fetches student profile from Supabase (name, preferred_country, education_level, field_of_study, ielts_score,
tone_preference). Retrieves RAG context filtered by country if available. Builds personalised prompt with both student
profile and package data injected. Calls Groq or Ollama. Returns reply text.

POST /upsell
Input: { phone_number, message_text }
Output: { upsell_message }
Fetches student profile and relevant package details from Supabase. Generates a reply that acknowledges the
student's current country interest, then naturally introduces a related package benefit or scholarship they may not have
asked about.
POST /churn-score
Input: { phone_number }
Output: { churn_score, risk_level }
Computes churn score from enquiry_events (abandoned_enquiries) and conversations table (message frequency drop).
Uses enquiry_events instead of cart_events — table name change from original schema.
GET /health
Output: { status: ok, llm_provider: groq|ollama }
Confirms the Flask server is alive and which LLM provider is active.
INTENT LABELS — STUDY ABROAD CONTEXT
package_enquiry
Student asking about a specific country package, fees, or services
visa_question
Questions about visa process, blocked account, APS, MVV, Campus France
scholarship_query
Asking about DAAD, Eiffel, Holland Scholarship, or other funding
complaint
Student is unhappy — delayed response, incorrect info, pricing concern
churn_risk
Signalling disengagement — 'I'm looking at other consultancies', long silence
general
Casual conversation, greeting, or unclassified message
CHURN SCORING FORMULA — UPDATED FOR STUDY ABROAD
score = (0.5 * days_inactive_norm) + (0.3 * abandoned_enquiry_rate) + (0.2 * message_drop_rate)
days_inactive_norm = min(days_since_active / 30, 1.0) abandoned_enquiry_rate =
abandoned_enquiries / max(total_enquiries, 1) # from enquiry_events table message_drop_rate = 1
- (last7 / max(prior7, 1)), clipped to [0, 1] Above 0.7 = high risk. 0.4 to 0.7 = medium. Below
0.4 = healthy. NOTE: 'abandoned_enquiries' replaces 'abandoned_carts' from the original schema.
Query: SELECT COUNT(*) FROM enquiry_events WHERE customer_id=X AND status='abandoned'
GROQ VS OLLAMA — HOW TO SWITCH
In your Flask app, the entire LLM call goes through one function. The only difference between Groq and
Ollama is the base_url and whether an API key is needed. LangChain's ChatOpenAI interface supports both.

Groq: base_url='https://api.groq.com/openai/v1', api_key=GROQ_API_KEY Ollama:
base_url='http://localhost:11434/v1', api_key='ollama' Set LLM_PROVIDER=groq or
LLM_PROVIDER=ollama in your .env file. Your Flask app reads this on startup and initialises the
correct client. One variable change, zero code change.
