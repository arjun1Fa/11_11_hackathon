Your Role in One Line
You are the nervous system of Smartilee. You connect WhatsApp to the AI brain via Happilee webhooks,
route every incoming student message to the right action, fire automated enquiry recovery and
re-engagement flows, and manage the human handoff process. You are the coordinator between Happilee,
the Flask AI backend, and Supabase.
HOW WHATSAPP INTEGRATION WORKS — HAPPILEE
Happilee is not a chatbot builder for Smartilee. We use Happilee only as a pipe — it gives us access to the
WhatsApp Business API. Here is the exact flow:
Step 1
Register your business WhatsApp number at happilee.io
Free sandbox available. This gives you a test number to send and receive WhatsApp messages.
Step 2
Give Happilee your Node.js server's public URL
Happilee will POST every incoming message to your webhook endpoint. Local testing: run ngrok http 3000
and give Happilee the ngrok HTTPS URL.
Step 3
Student sends a WhatsApp message
Happilee receives it and immediately forwards a JSON payload to your webhook endpoint.
Step 4
Your Node.js server processes the message
Calls the Flask AI backend, gets a reply decision, generates the reply, and sends it back through Happilee's
outbound REST API.
Step 5
Template messages for proactive outreach
For messages sent outside the 24-hour window (enquiry recovery, re-engagement), WhatsApp requires
pre-approved message templates. Register these in Happilee.

Content-Type: application/json } Body (text message): { to, type: "text", text: { body: "..." }
} Body (template — for enquiry recovery / re-engagement): { to, type: "template", template: {
name: "package_followup", params: ["Priya", "Germany"] } }
MESSAGE ROUTING LOGIC
After calling POST /analyze on the AI backend and receiving the action field:

· auto_reply → call /generate-reply, get reply_text, send via Happilee sendMessage()
· upsell → call /upsell, get upsell_message, send via Happilee sendMessage()
· handoff → handleHandoff(phone_number, reason) — insert to handoff_queue, notify owner, stop AI replies
· schedule_followup → insert row into followups table, cron job will send it later

KEY SCHEMA CHANGE — ENQUIRY_EVENTS REPLACES CART_EVENTS
The only structural change from the original Smartilee schema that affects your code is the rename of
cart_events to enquiry_events. The table structure and logic are identical. Update all references in your
code:
// OLD (e-commerce): supabase.channel('cart-events').on('postgres_changes', { table:
'cart_events', filter: 'status=eq.abandoned' }, handleAbandon) // NEW (study abroad):
supabase.channel('enquiry-events').on('postgres_changes', { table: 'enquiry_events', filter:
'status=eq.abandoned' }, handleEnquiryAbandon) // handleEnquiryAbandon sends a package
follow-up instead of a cart recovery message: sendTemplate(phone, 'package_followup',
[customer.name, customer.preferred_country]) // UPDATE enquiry_events SET
status='followup_sent', updated_at=now()
Also update the outbound Supabase log call to include the new package_context field when logging
conversations:

// Log inbound/outbound — add package_context from /analyze response: await
supabase.from('conversations').insert({ customer_id, direction, message_text, intent_label:
analyzeResult.intent, sentiment: analyzeResult.sentiment, action_taken: analyzeResult.action,
language: analyzeResult.language, package_context: analyzeResult.preferred_country // NEW FIELD
})

PROMPT 2 — Webhook + Full Routing (Study Abroad)
In my Smartilee Node.js server for a study abroad consultancy, build webhook.js and add route to server.js. POST
/webhook must: 1. Parse Happilee payload: { from, message: { text }, timestamp }. phone_number = from,

'handoff' -> call handleHandoff(phone_number, 'ai_escalation'). 'schedule_followup' -> INSERT into followups with
package_id from customer profile, skip outbound. f. For auto_reply/upsell: sendMessage(phone_number, reply_text).
Log outbound to conversations. try/catch at every step. console.error on failures.
